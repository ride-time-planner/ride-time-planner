# ルート走破時間シミュレータ — 設計書

作成: 2026-07-08 / 対象: HTML+JS 単体ツール（将来は静的ホスティング）

---

## 1. 目的とスコープ

RideWithGPS 相当の地図・標高プロファイルを表示し、**開始時刻**と**各地点の滞在時間**を入力すると、
「何キロ地点に何時頃に到達するか」を標高グラフ上のマウスオーバーで確認できるツール。
走破時間は 2 つの推定モデル（簡易 / パワー物理）を切り替え可能とし、VAM 等の指標も併記する。

本フェーズの成果物は **設計 + 動く骨組み（スケルトン）**。
GPX 読み込み・時刻計算・VAM 算出・localStorage 保存までは実際に動く状態で提供し、
RWGPS URL 取り込みは「抽象化した仕組み＋プロキシ設定欄」を用意し、GPX を確実なフォールバックとする。

---

## 2. 全体アーキテクチャ

サーバーを持たない静的構成（HTML/CSS/JS のみ）。外部依存は CDN の地図ライブラリのみ。

```
index.html
├─ css/styles.css
├─ js/
│  ├─ app.js        … 画面初期化・イベント配線・全体オーケストレーション
│  ├─ state.js      … localStorage への保存/復元（状態の単一の窓口）
│  ├─ route.js      … ルート取り込み（GPX パーサ / RWGPS ローダ）＋距離・勾配の前処理
│  ├─ import.js     … Strava 実走(GPX/TCX/ZIP)取り込み・スクリーニング・簡易モデル校正
│  ├─ models.js     … 速度モデル（簡易 / パワー物理）＋ VAM 等の指標計算
│  ├─ schedule.js   … 開始時刻＋滞在時間から各点の到達時刻を計算
│  ├─ map.js        … Leaflet 地図描画・ルートライン・カーソル連動マーカー
│  └─ chart.js      … 標高プロファイル描画（canvas）・ホバーツールチップ
└─ worker/
   ├─ worker.js     … RWGPS 公開ルート用 CORS プロキシ (Cloudflare Workers)
   └─ README.md     … Worker デプロイ手順
```

CDN 追加: JSZip（Strava 一括 ZIP 解凍時のみ遅延ロード）。

依存ライブラリ（CDN）:
- **Leaflet** + OpenStreetMap タイル … 地図（無料・静的可）
- 標高グラフは **canvas 自前描画**（依存を増やさず、ホバー時刻表示を細かく制御するため）

データフロー:

```
ルート入力(GPX/URL) ─▶ route.js: 点列[{lat,lng,ele,dist}] + 区間勾配
        │
        ├─▶ map.js:    地図にライン描画
        ├─▶ models.js: 区間ごとの速度 → 区間所要時間
        │        ▲ プロファイル(FTP/体重/CdA/簡易テーブル)
        └─▶ schedule.js: 開始時刻 + 滞在時間 → 各点の到達時刻(累積)
                 │
                 ▼
        chart.js: x=距離 / y=標高、ホバーで「◯km / 標高◯m / 勾配◯% / 到達 ◯:◯」
```

## 3. RideWithGPS URL 取り込みの方針（重要）

**課題**: 静的サイトからブラウザで RWGPS のルートデータ（`.../routes/{id}.gpx` や API の `.json`）を
直接 `fetch` するとクロスオリジン（CORS）でブロックされる。RWGPS はブラウザ直叩きを許可していない。

**採用方針（3 段構え、RouteLoader で抽象化）**:

1. **プロキシ経由（本命）**: 軽量プロキシ（例: Cloudflare Workers）を 1 つ挟み、
   `https://<自分のworker>/rwgps?id=48058893` → Worker 側で RWGPS からルートを取得し CORS ヘッダ付きで返す。
   将来の静的ホスティングと相性が良く、URL からのワンクリック取り込みを実現できる。
   本フェーズではフロント側に「プロキシURL設定欄」を用意し、未設定でも動くようにする（Worker 実装は次フェーズ）。
2. **GPX アップロード（確実なフォールバック）**: RWGPS/Strava から書き出した GPX をドラッグ&ドロップ。
   サーバー不要・オフライン可で最も確実。**初期リリースの主軸**。
3. **API + トークン**: RWGPS API キー/認証を使う方式。CORS は依然プロキシが必要なため、1 の内側の実装差分として扱う。

URL からルート ID を抽出する処理は route.js に実装済み（プロキシが無い場合は GPX 導線を案内）。

## 4. 速度モデル（2 つ切替）＋ 指標

区間 i（距離 d_i, 標高差 h_i, 勾配 g_i = h_i/d_i）ごとに速度 v_i を求め、時間 t_i = d_i / v_i を積算する。

### モデル A: 勾配-速度テーブル（簡易）
平地基準速度 `Vflat` に勾配補正をかける直感的モデル。設定項目が少なく、まず動かす用途。

```
下り(<-4%)      : Vflat * 1.4（上限あり）
ゆる下り(-4〜-1%): Vflat * 1.15
平地(-1〜1%)    : Vflat
ゆる登り(1〜4%) : Vflat * 0.7
登り(4〜8%)     : Vflat * 0.45
急登(>8%)       : Vflat * 0.30
```
（係数はプロファイルで調整可能にする）

### モデル B: パワー物理モデル
一定パワー `P`（FTP × 目標強度）で走ると仮定し、各区間で力の釣り合いから v を解く。

```
P = v * ( F_roll + F_grav + F_aero )
  F_roll = Crr * m * g * cosθ
  F_grav = m * g * sinθ
  F_aero = 0.5 * ρ * CdA * v²        （向かい風は将来拡張）
→ 各区間で v の 3 次方程式を数値解法（二分法）で解く
```
パラメータ: `FTP`, 目標強度(%), `m`(=体重+機材重量), `CdA`, `Crr`, 空気密度 `ρ`。
Zwift の zFTP/体重を初期値として流用できるよう入力欄を用意。

### 出力指標
- 総所要時間 / 走行時間（滞在を除く） / 到着予定時刻
- 平均速度
- **VAM**（登坂のみの平均上昇速度 = 上昇標高 / 登坂時間 × 3600, m/h）… 全体
- 獲得標高 / 距離 / 平均勾配・最大勾配
- （パワーモデル時）平均パワー・推定 W/kg

## 5. localStorage スキーマ

キーは `rts.` プレフィックス。単一プロジェクト構成（将来は複数ルート対応）。

```json
{
  "rts.route": {
    "name": "…",
    "source": "gpx | rwgps",
    "points": [{"lat":0, "lng":0, "ele":0, "dist":0}]
  },
  "rts.plan": {
    "startTime": "07:00",
    "stops": [ {"distKm": 25.3, "durationMin": 15, "label": "補給"} ]
  },
  "rts.profile": {
    "model": "simple | power",
    "flatSpeed": 25,
    "ftp": 250, "intensity": 0.75, "riderWeight": 68, "bikeWeight": 9,
    "cda": 0.32, "crr": 0.005, "rho": 1.225
  },
  "rts.settings": { "proxyUrl": "" }
}
```

保存は state.js が一元管理（`State.save()/load()`）。入力変更時に自動保存。

## 6.5 入力取り込みの実装（追加）

- **RWGPS URL**: 公開ルートの GPX を Cloudflare Worker プロキシ経由で取得（`worker/`）。設定のプロキシURLに `.../rwgps?id={id}` を登録。
- **Strava 実走（校正）**: GPX/TCX/ZIP を複数取り込み。除外距離でスクリーニング後、約100mウィンドウで勾配別に速度中央値を算出し、
  平地速度＋勾配バケット係数を求めて簡易モデルに反映（`profile.flatSpeed`, `profile.factors`）。空バケットは既定係数へフォールバック。
  ZIP は代表60件までを標本化（大容量対策）。時刻(`<time>`)の無いトラックは速度算出不可のためスキップ。
- **Zwift 指標**: zFTP/zMAP/VO2max を手動入力（エクスポート手段が無いため）。zFTP はパワーモデルの FTP に反映。走行データ自体は Strava 連携で流入。

## 6.6 追加実装（第2弾・実装済み）

- **標高スムージング**: 距離ウィンドウ±120mの移動平均で GPS 標高ノイズを低減（route/実走とも）。勾配・VAM・プロファイル精度が向上（検証: 隣接標高差の総和 1200→107）。
- **クライム別VAM**: 主要クライムを自動検出（獲得≥100m・長さ≥1km・平均勾配≥3%、小さな下り<25mは連続扱い）。距離/獲得/平均勾配/VAM/取付通過時刻をサマリに上位5件表示。
- **Strava一括ZIP**: `.gz`(gpx/tcx) を DecompressionStream で解凍。`activities.csv` を読み、Activity Type がライド系のもののみ・名前に除外キーワード(既定「通勤」)を含むものを除外。代表80件までを標本化。`.fit`(バイナリ)は非対応でスキップ（次フェーズ）。
- **向かい風（パワーモデル）**: 風速＋風向（吹いてくる方位）を入力。区間ごとの進行方位との角度から向かい風/追い風成分を求め、空気抵抗を対気速度(v+風成分)で計算（検証: 平地 向22.1<無33.1<追46.6 km/h）。
